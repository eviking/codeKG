# Agent Index System

> The agent index is codeKG's primary output — a directory of pre-computed markdown files committed into the repo that give AI coding agents complete structural knowledge without requiring any source file exploration.

---

## The core idea

Every time an agent needs to understand a codebase, it has two options:

1. **Explore** — use `find`, `Read`, `grep`, and `ls` to discover structure organically. This costs 20–40 tool calls, produces partial understanding, and is repeated identically by every future session.

2. **Read the index** — read `.codekg/INDEX.md` (3 seconds), identify the relevant module file, read it (5 seconds). Complete understanding of every class and method in the module.

The agent index makes option 2 always available and always current.

---

## File structure

After a publish, `.codekg/` in the repo looks like this:

```
.codekg/
├── INDEX.md                          # master navigation — read this first
├── architecture/
│   ├── datastores.md                 # all data stores with schemas
│   ├── dependencies.md               # cross-module import graph
│   ├── hotspots.md                   # highest blast-radius classes
│   ├── modules.md                    # module map with class counts
│   ├── patterns.md                   # architectural patterns detected
│   ├── schema.md                     # Neo4j node labels and properties
│   └── violations.md                 # current policy violations
├── insights/
│   └── index.md                      # all captured insights, grouped by module
├── modules/
│   ├── services--api.md              # full class+method detail for services/api
│   ├── services--console.md          # ...
│   ├── services--ingestion.md        # ...
│   ├── services--mcp.md              # ...
│   └── services--watcher.md          # ...
└── policies/
    └── active.md                     # active architectural policies
```

> For repos under 2,500 total LOC, a single `modules/combined.md` is generated instead of per-module files — all modules inlined in one file so the agent never needs to decide which to read.

## File generation

Each file is generated by a dedicated function in `services/api/agent_index/generator.py`:

| File key | Generator function | Cap | Output location |
|----------|-------------------|----|----------------|
| `INDEX.md` | `generate_index()` | 8K chars | `.codekg/INDEX.md` |
| `architecture/modules` | `generate_modules()` | 12K chars | `.codekg/architecture/modules.md` |
| `architecture/dependencies` | `generate_dependencies()` | 12K chars | `.codekg/architecture/dependencies.md` |
| `architecture/hotspots` | `generate_hotspots()` | 12K chars | `.codekg/architecture/hotspots.md` |
| `architecture/patterns` | `generate_patterns()` | 10K chars | `.codekg/architecture/patterns.md` |
| `architecture/violations` | `generate_violations()` | 10K chars | `.codekg/architecture/violations.md` |
| `architecture/datastores` | `generate_datastores_index()` | 40K chars | `.codekg/architecture/datastores.md` |
| `architecture/schema` | `generate_schema()` | 10K chars | `.codekg/architecture/schema.md` |
| `policies/active` | `generate_policies()` | 10K chars | `.codekg/policies/active.md` |
| `insights/index` | `generate_insights_index()` | 10K chars | `.codekg/insights/index.md` |
| `modules/<name>` | `generate_module_file()` | 60K chars | `.codekg/modules/<name>.md` |
| `modules/combined` | `generate_combined_index()` | 120K chars | `.codekg/modules/combined.md` |
| `claude_md` | `generate_claude_md_snippet()` | — | `CLAUDE.md` (repo root) or `.codekg/CLAUDE.md` |
| `agents_md` | `generate_agents_md_snippet()` | — | `AGENTS.md` (repo root) or `.codekg/AGENTS.md` |

All generators query Neo4j directly using the module-level `run_query` function. They never read source files.

> **`claude_md` vs `agents_md`:** These two keys generate different content. `claude_md` targets Claude Code — it references MCP tools (`capture_insight`, `get_change_impact`, `search_classes`). `agents_md` targets Codex / OpenAI agents — it uses shell commands (`cat .codekg/INDEX.md`) since Codex has no MCP support. Both can be published to the same repo simultaneously.

## The status lifecycle

```
[new repo]
    │
    ▼
upsert_file()  →  status='current', published_at=NULL
    │
    ▼
mark_published()  →  status='current', published_at=<ts>, published_sha=<sha>
    │
    │  [regen runs again, content changes]
    ▼
upsert_file()  →  status='stale', published_at=<ts> (preserved)
    │
    │  [publish runs]
    ▼
mark_published()  →  status='current', published_at=<new ts>, published_sha=<new sha>
```

**Status values:**
- `current` — content matches what's published in the repo
- `stale` — content has changed since last publish (file shows ⚠ in UI)
- Hidden files are excluded from publish entirely

## Empty file detection

After generation, each file is checked for meaningful content:

```python
def _content_is_empty(content: str) -> bool:
    body_lines = [l for l in content.splitlines()
                  if l.strip() and not l.startswith("#") and not l.startswith("_Generated")]
    body = "\n".join(body_lines).strip()
    _EMPTY_PHRASES = ("_No ", "not found", "No insights recorded",
                      "No violations detected", "No active policies")
    return len(body) < 80 or any(p in body[:200] for p in _EMPTY_PHRASES)
```

Empty files are automatically hidden. A hidden file:
- Shows with an 🈳 "empty" pill in the console
- Is excluded from publish
- Has its corresponding `.codekg/` file deleted on next publish (ghost file cleanup)
- Is NOT referenced in INDEX.md

This prevents the INDEX.md from pointing agents at files that contain no useful information.

## Ghost file cleanup

When a file is hidden (manually or auto-detected as empty), the next publish run deletes its on-disk counterpart:

```python
# In _ai_publish():
expected_paths = {repo_path / ".codekg" / f["directory"] / f["filename"]
                  for f in visible_files}

# Delete any .codekg/ file not in expected set
for p in codekg_dir.rglob("*"):
    if p.is_file() and p not in expected_paths:
        p.unlink()
        deleted.append(str(p.relative_to(repo_path)))
```

This handles:
- Files hidden via the toggle button
- Files auto-hidden as empty
- Files removed from `_INDEX_FILES` in a code update
- Old per-module insight files after the consolidation to `insights/index.md`

> `CLAUDE.md` and `AGENTS.md` at the repo root are **not** managed by ghost cleanup — they live outside `.codekg/` and are never deleted by this loop.

## Manual additions

Each file has an optional `manual_additions` field. Content stored here is appended to the file on publish, below the auto-generated content. It survives regeneration — the KG-generated content is replaced but manual additions are preserved.

Use this for:
- Architectural decisions that aren't derivable from the graph
- Known gotchas that can't be captured as insights
- Links to related documentation

```
# From the console file detail page:
manual_additions = """
## Known limitation
The `wire_edges` timeout is set conservatively at 90s. For repos with >10k classes,
consider running ingestion in two passes: parse first, then wire_edges separately.
"""
```

## Regeneration trigger

The watcher calls `POST /agent-index/regen` after a successful ingestion scan:

```python
# In services/watcher/main.py, after scan container exits:
api.post("/agent-index/regen", json={"repo_id": repo_id})
api.post("/agent-index/publish", json={"repo_id": repo_id})
```

This means `.codekg/` is updated automatically on every commit — no manual intervention needed once the repo is registered.

## Console management

The console at `http://localhost:8080/agent-index` provides:

- **Card grid** — one card per file, showing status badge, char count, published/stale indicator
- **Regenerate all** — triggers full regen for the selected repo
- **Publish to repo** — writes files to disk and git commits
- **Per-module table** — shows status for each module's index file
- **File detail view** — preview rendered markdown, regenerate, hide/show, edit manual additions

Status badges:
- 🟢 `current` — generated and published, content matches
- 🟡 `stale` — regenerated after last publish, needs republish
- ⚠ `needs republish` — same as stale but shown next to published date
- 🈳 `empty` — auto-hidden, no meaningful content
- 🙈 `hidden` — manually hidden

## Adding a new index file

```python
# 1. Write the generator function in generator.py
def generate_my_new_file(repo_id: str) -> str:
    rows = run_query("MATCH ... RETURN ...", repo_id=repo_id)
    lines = [f"# My File — {repo_id}", f"_Generated {_TS()}_", ""]
    for r in rows:
        lines.append(f"- `{r['name']}` — {r['description']}")
    return _cap("\n".join(lines), 10_000)

# 2. Register it in _INDEX_FILES at the bottom of generator.py
{
    "key":         "my-area/my-file",
    "directory":   "my-area",
    "filename":    "my-file.md",
    "description": "One-line description for INDEX.md",
    "trigger":     "any_change",
    "generator":   generate_my_new_file,
},

# 3. Add the cap in _CAP dict if it needs a custom limit
_CAP["my_file"] = 8_000

# 4. Rebuild api container and regen
docker compose up -d --build api
POST /agent-index/regen {"repo_id": "codeKG"}
POST /agent-index/publish {"repo_id": "codeKG"}
```
